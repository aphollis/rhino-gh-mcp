# Efficiency & Token-Reduction Build Plan

Goal: make the agent fast and cheap to run inside Rhino, and stop it writing
bespoke scripts for every action. Sequenced by ROI and dependency.

## The three moves

Every lever below is one of:

1. **Cache** — put static content in the prompt-cached prefix so it's paid once.
2. **Off-budget** — move work (layout, validation, diffing, param resolution)
   onto the server so the model never spends tokens on it.
3. **Shrink** — reduce the identifiers/payloads the model carries turn-to-turn
   (handles not GUIDs, summaries not dumps, images not coordinates).

## The token sinks we're attacking (from the live-test trace)

For a trivial slider→circle the agent did 6 round-trips + a confused duplicate:
`ToolSearch → gh_launch → gh_build_recipe ✓ → gh_build_recipe (GUIDs-as-keys, confused) → ToolSearch → gh_connect`.

| # | Sink | Root cause |
|---|------|-----------|
| S1 | GUID juggling | 36-char IDs repeated every call; caused the duplicate-build confusion |
| S2 | Schema-load round-trips | Deferred tools need a `ToolSearch` before first use |
| S3 | Discovery | `gh_search_components` + `gh_component_info` before every wiring |
| S4 | Full canvas dumps / Python scripts | Unbounded payloads; the fallback when no dedicated tool exists |

---

## Phase 0 — Instrument first (measure, don't guess)

Can't prove a lever works without a baseline.

- Per-request logging in `agent/server.mjs`: turns, tool-call count, input/output
  tokens, cache-read tokens, `total_cost_usd`. Append to `agent/metrics.jsonl`.
- A tiny `bench/` script that runs a fixed set of prompts ("build a tower",
  "voronoi panel", "diagnose this canvas") and reports avg turns/tokens/cost.
- Run it once now → this is the number every later phase is measured against.

**Effort:** S · **Depends on:** nothing · **Serves:** all

---

## Phase 1 — Cacheable static context

Amortize discovery and schema loading to ~0.

- **Component reference** (`docs/gh_reference` generated from the live library):
  compact table of the ~80 most-used components with exact input/output param
  names + the input primitives. Generated by a one-off `rhino_execute_python`
  sweep of the ComponentServer, checked into the repo.
- Inject it into the **system-prompt append** (cached position) — and also expose
  it as an **MCP resource** so the client caches it instead of re-sending.
- **Preload the core 6–8 GH tools** (launch, build, edit, canvas, output, bake,
  search, info) so everyday work skips `ToolSearch`.
- Audit the server so *all* static content (system prompt, reference, schemas)
  lands in the cacheable prefix.

**Attacks:** S2, S3 · **Effort:** M · **Depends on:** 0 · **Serves:** efficiency

---

## Phase 2 — Stable handles + terse, self-correcting returns

Kill GUID bloat and the confusion it causes.

- Listener keeps a per-document **`key → GUID` map**. `gh_build_recipe` / `gh_add`
  return short keys (`r`, `c`); `gh_connect` / `gh_set_value` / `gh_get_output` /
  `gh_bake` accept keys as well as GUIDs.
- **Terse return contract:** success returns the minimum (keys + healthy flag);
  on error return the **available param names + nearest match** so the model
  self-corrects in the same turn instead of calling `gh_component_info`.
- Declare **output schemas** on the tools so the model can predict shapes.

**Attacks:** S1, S3 · **Effort:** M · **Depends on:** 1 · **Serves:** efficiency + correctness

---

## Phase 3 — Idempotent builds + off-budget verify + lean reads

Move the verify loop and read cost off the token budget; fix the duplicate bug.

- **Idempotent keyed builds:** re-running `gh_build_recipe` with the same keys
  *updates* existing components instead of duplicating. (Would have made the
  confused second build a no-op.)
- **Server-side validation:** build/edit return `"healthy"` or a targeted problem
  list; the model only re-engages when something is actually wrong.
- **`gh_get_canvas` detail levels:** default = one line per object (key, type,
  error flag); full dump only on request or only for errored nodes.
- **Semantic diff:** server tracks a canvas version; reads return only what
  changed since the last read.
- **`gh_edit`** batch tool: many set/connect/disconnect/delete ops in one call.

**Attacks:** S1, S4 · **Effort:** M–L · **Depends on:** 2 · **Serves:** efficiency + correctness

---

## Phase 4 — Reuse layer (the "recipes as product" play)

Stop regenerating common graphs from scratch.

- **Template library:** `gh_list_templates` / `gh_apply_template` with proven
  parametric definitions (tower, louver facade, voronoi panel, waffle, diagrid…)
  and exposed parameters. Agent picks one and sets params → near-zero tokens,
  reliable output.
- **`.gh` snippet library** on disk: merge proven subgraphs by name
  (attractor field, panel-with-thickness) via `gh_open`/merge.
- **Auto-cluster** repeated motifs so later references/reads are one node.
- Expose recurring workflows ("build parametric X", "diagnose errors", "expose
  these as sliders") as **MCP Prompts**.

**Attacks:** S4 (biggest long-term) · **Effort:** L · **Depends on:** 2, 3 · **Serves:** templates/product

---

## Phase 5 — Agent-loop routing & richer returns

Per-turn cost and long-session context control.

- **Auto model routing:** Haiku for mechanical/discovery/summary steps, escalate
  to Opus/Fable only for design reasoning. Cheap planner turn emits a recipe the
  expensive model never hand-assembles.
- **GH-builder subagent:** isolated context does the build, returns a short
  result; the main conversation never carries the full tool transcript.
- **Prefer image returns:** rendered canvas/geometry thumbnails over coordinate
  dumps — cheaper and more useful.

**Attacks:** per-turn cost, context growth · **Effort:** M · **Depends on:** 1–3 · **Serves:** efficiency + polish

---

## Implementation status (2026-07-17)

Phases 0–5 implemented and committed. Verified where possible without a full
live Rhino session (metrics baseline captured; `alwaysLoad`/no-ToolSearch
confirmed live; listener syntax-checked with Rhino's CPython 3.9; template and
routing logic unit-checked). Full end-to-end live verification happens at the
plugin rebuild.

- **Subagent routing (Phase 5)** was deferred: the in-Rhino panel doesn't surface
  MCP prompts or Task delegation, and spawning subagents that re-open the MCP
  server adds risk for little gain at current session sizes. Revisit if sessions
  grow long. Auto model routing and image returns cover the rest of Phase 5.
- **MCP Prompts (Phase 4)** likewise deferred — same reason (custom panel, not a
  generic MCP client). Templates deliver the reuse win instead.

## Recommended sequence

`0 → 1 → 2 → 3 → 4 → 5`.

Phases 1–3 are the expanded "efficiency trio": cheap, foundational, and they
attack the exact waste in the trace. Phase 4 is the template/product direction,
built on the leaner primitives. Phase 5 is routing polish. Phase 0 is one small
session up front so every later claim is measured, not asserted.
